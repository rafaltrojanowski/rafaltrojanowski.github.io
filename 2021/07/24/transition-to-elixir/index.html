Transition to Elixir

Jul 24, 2021

As a Ruby on Rails Developer, I’ve heard about Elixir/Phoenix and its similarities. I wanted to expand my skill set, and learn something new. Functional programming sounded attractive, that’s why I chose Elixir. I started using Elixir as a Backend language professionally 6 months ago.

The few first months of learning Elixir and functional programming were about writing small prototypes. Sometimes it was a struggle because it was a solo journey, but it was worth doing exercise, just to explore the ecosystem. Later on, I’ve joined Slack group and started asking questions, when I’ve been stuck. Folks in the community are helpful and friendly. They helped to clarify thoughts, and get moving again. It was a great experience.

Joining an Exercism helped me to speed up with grasping concepts of functional programming and solve more advanced problems. It was quite good to have an access to the solutions of experienced developers from which I could learn how I could write better code. This path of learning gave me the most fun.

Later on, VSCode extension ElixirLS, with autocomplete, jump to function definition, and so on, was super handy. Compiler warnings/errors (warnings of undefined functions) have been an enormous help. Compared to Ruby, where you get errors at runtime, it has been an advantage. In general, less magic is happening in Elixir/Phoenix, which means fewer small changes in how magic behaves to bite you.

Elixir does not have so many libraries as Ruby, but those that are available are mature. Sometimes you will need to build a native API client from scratch, but it should not be difficult. You will get productive quickly with Ruby background. If you need concurrency and performance, Elixir would be a great option.

For anyone interested in learning Elixir, I’d recommend thinking of a project that excites you, rather than doing tutorials. By having a goal to build a prototype, you will be not bored as you would by copy-pasting the code.
