Vim`s global search and replace

I have been looking for efficient solution how to refractor code with Vim for a while,
so I’d like to share it with you. That’s an extremely easy:

  1. Search (Tool from Vim):

:Ack OldClassName **/*.rb **/*.rake

  1. Apply changes on results (optional):

:cdo g/OldClassName/s/OldClassName/NewClassName/ge

  1. Update and save at once:

:cdo %s/OldClassName/NewClassName/ge | update

Big thanks to Sebastian for sharing this trick with me.

Check out his blog post Should You start using VIM?

Big up!

Redirect back with Ember.Location

Motivation

In the approach with routable modal described in previous posts, there is one drawback. Closing link located in the modal need to be handled with additional logic if we want to achieve the same effect when we don’t use routable approach. In routable approach we open modal for a specific route so close action need to make a redirect to the previous route from the history.

Basically we can handle closing modal by simple link-to pointing to the main application route.

But what when we want to redirect user to the previouse view, similarly like when user click on back button in the browser?

There is a problem how to track history of visited routes. I was researching a long time and found a few solutions:

  1. Use deditated add-ons:

    • ember-cli-history-mixin
    • ember-route-history

    Disadvantage: Seems to be not maintaned and problems with recording history of nested routes. Generally fetching ids from the router object is hard.

  2. Use window.history.back()

    That looks fine, but doesn’t work the same way in all browsers. Also, it would redirect user outside the app.

  3. Some code snippets on Stackoverflow which hacks router to extract previouse route state.

    Code seems quite complicated and hacky.

Finally I chose solution on this blog: http://kiprosh.com/blog/ember-js-how-to-track-last-visited-route

Although, it wasn’t perfect, because it still didn’t work with nested routes.

On the occassion, I approached it with TDD beliving that I will discover something new.

…and started debugging and looking at router object, to retrieve some info current route path.

Happily, I found something new - router.location.path which stores the string. I was suprised. It looks like something new in Ember and don’t need to hack it to make a transition back.

So updated recording last visted route in beforeModel hook:

routes/search.js

routes/search.hbs

controllers/search.js

I was happy that I finally found solution for problem I was struggling long time. However I discovered something unexpected when I tested app manually!! That didn’t work. Again using debugger I investigated that path is undefined and router has problem with transition. Say what!? Tests are green.

Ember.Location API

I learned that Ember uses different strategies for routing location in test and development mode by default. You can see it when you look at config/environment.js

Started diggin in the docs (https://www.emberjs.com/api/ember/2.14.1/namespaces/Ember.Location) and source code to find more.

Generally, there are 4 types of location that can be used by Ember:

Ember.NoneLocation

It’s used in tests. That makes sense because when test are running there is url like:
I switched to this type in the development and then router.location.path was again accessible and redirection worked as expected. But url was blank all the time, so I had to gave up this solution.

That type could be also usable when you embeed your Ember application in a larger page.

Ember.HashLocation

This is default. The characteristic thing is that it adds # on the the begining of path in the url.
I noticed that in the code introduced router.location.lastSetPath which sounds familiar and contains the same string with full path like router.location.path in Ember.NoneLocation.

Note: the value is set only when transition is performed by router. In other words it won’t work on copy paste an url and hit Enter.

Ember.HistoryLocation

Using HistoryLocation results in URLs that are indistinguishable from a standard URL. This relies upon the browser’s history API.

Ember.AutoLocation

Using AutoLocation, the router will use the best Location class supported by the browser it is running in.

Desing Fullscreen Search Interface With Ember - Part 1

Motivation

Some time ago I had been working on a search feature. The UI was designed in a fullscreen page.
As hiding nabvar and footer or providing separate layout was tricky and not intuitive I decided to use fullscreen modal dialog.

Generally, modal dialogs have become a ubiquitous feature of mobile web design, but designers need to put more thought into when to use them, and when to leave them.
You can read more here: https://uxmag.com/articles/modals-on-mobile-how-to-use-them-wisely

By the way, you would ask why designer planned interface as a full page view?

From the usability perspective modal rendered in fullscreen have a few advantages:

  • it keeps user focus only on particular task. We don’t want to distract the user with the content “behind” the modal
  • it will work on mobile devices as modal size will adapt to every screen size
  • there are opinions that fullscreen modals look better and cleaner

Summing up requirements are as following:

  • Implement modal dialog responsible for data seaching;
  • Render modal dialog in fullscreen;
  • Modal dialog should be accessible from everywhere in the application;
  • It should be possible to close search modal dialog and bring back user to the home view.

We need to keep in mind a few things:

  • Users will hit the back button in the browser when modal is opened;
  • Process of loading data asynchronously should be done in “background”, we want to show render modal smoothly
  • URL should store state. Thanks to this, it will work nicely with user refreshes the page.
  • We want to stick to Ember convention (route based model fetching) to prevent problems in future ***
